01
What we observe Symptoms in the wild

Cold start works once

Each fresh notifier processes exactly one event after launch, then goes silent. The very first tts+banner entry is logged within ~5s of the listener's start line — and nothing follows.

Bytes flow but stay buffered

netstat shows the listener's curl received 4559+ bytes over its lifetime — roughly eleven events worth of SSE traffic — none of which ever reach _process_event.

Fresh subscribers see everything

A parallel curl spawned from the shell receives every event in real time, while the long-lived listener's curl receives the same bytes but its bash read never advances.

No errors, no dedup, no skip

No dedup skipped, no unknown channel, no stack trace. Pure silence — which is the most diagnostic signal of all.

02
Why it happens The bash 3.2 process-substitution race
nexus-agent SSE @ :7400
tcp · 4559 b
curl --no-buffer child pid 3604
pipe · 16 KiB
bash 3.2 · while read stalled after evt #1
Mechanism (high confidence). The script's listener uses done < <(curl ...) — a process substitution. Each iteration calls _dispatch_banner, which spawns terminal-notifier or osascript as a child. In bash 3.2 (shipped 2007, the last GPL-2 release before Apple froze it), spawned children inherit the curl pipe's read FD. Until that child fully exits and bash reaps it, the kernel keeps the FD open with two readers — and bash's outer read can deadlock against it. The pipe fills, curl blocks on write, the listener appears alive but processes nothing.
03
Two ways out Option A vs Option B
Option A · root

Replace the shell.

Same script. Modern bash. The race goes away.

The change

// ~/Library/LaunchAgents/com.nexus.notifier.plist
<key>ProgramArguments</key>
<array>
- <string>/bin/bash</string>
+ <string>/opt/homebrew/bin/bash</string>
  <string>-c</string>
  <string>~/bin/nexus-notifier.sh listen</string>
</array>

Why it works

  • FD inheritance fixed. bash 5.x propagates close-on-exec correctly to children — no more shared read end on the curl pipe.
  • Untouched logic. The 646-line listener script is unchanged. No risk of breaking other invariants.
  • One-line plist edit. Reverts in 30 seconds if it misbehaves.

What it costs

  • Local-only. The fix lives on this Mac; a fresh checkout on the homelab still uses /bin/bash.
  • Adds an implicit dependency on Homebrew bash being installed at /opt/homebrew/bin/bash.
  • Doesn't help if the upstream script ever picks up its own #!/bin/bash shebang from another caller.

Effort

  • ~2 minutes. Edit plist, launchctl bootout / bootstrap, fire test event, watch the log.
Option B · safety-net

Watch the watcher.

Accept the bug. Detect stalls. Recycle the pipe.

The change

// new — managed by chezmoi
~/.local/bin/nexus-notifier-watchdog

# every 90s, if no log entry in 5 min →
# SIGTERM the curl child; outer loop reconnects.

find $LOG -mmin -5 -size +0c | grep -q . \
  || pkill -f 'curl.*events/stream'

Why it works

  • Self-healing. The notifier's outer while true; _run_stream; sleep 5; done reconnects after curl dies — restoring service within ~5 min of any stall.
  • Lives in the dotfiles repo. Same fix on Mac (launchd) and Arch (systemd timer). Reproducible across both machines.
  • Handles other stall modes too. Network drops, half-closed sockets, future bash bugs — anything that produces "log goes quiet" gets healed.

What it costs

  • Up to 5 minutes of dropped notifications per stall — the detection window.
  • Adds a third moving piece (watchdog) on top of agent + listener.
  • Treats the symptom, not the cause. The race is still there.

Effort

  • ~30 minutes. Script, plist/timer, chezmoi onchange hook, soak test on both OSes.

The honest summary. Option A is a fix: cheap, durable, and surgical, but parochial — it lives only on this Mac and depends on the upstream script not regressing. Option B is a policy: it doesn't fix the bug, but it institutionalizes recovery from any stall (this one and others), and it ships through the same chezmoi pipeline that already governs both machines. They're not mutually exclusive — A on its own is the highest leverage; A+B is the bulletproof posture.

Right now the cleanup banners fire natively via osascript and don't depend on this listener at all. That means the question is no longer "does cleanup work". It's "do you want every other tool that posts to nexus to recover automatically when the listener stalls?"

if yes → option B (chezmoi-managed) · if "fix it for me on this mac and we'll deal with arch later" → option A (one plist line)

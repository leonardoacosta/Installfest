What does ~10s
convergence mean?

Convergence is the time from "file changes on machine A" to "machine B has the same byte-identical file." With Syncthing this is not instant — there's a deterministic five-step pipeline. Most of the time, it completes in roughly ten seconds on a LAN.

02 / Power-loss scenarios
cut at t = 0.5s

Early cut

Homelab fsync'd the JSONL but Mac hasn't seen the announce yet. Message is stranded on a dead box. When power returns, Syncthing resumes and the lost interval syncs. Until then, you resume from the Mac's last-converged state.

Lose 1 in-flight msg
cut at t = 8.0s

Late cut

Mac has already pulled all blocks and may have completed rename. The new version exists on the source-of-truth machine. Resume on Mac is exact: the conversation continues with zero data loss.

No data lost
cut mid-rename(2)

Atomic guarantee

rename(2) on POSIX is atomic by spec. Either the old inode survives or the new one does — never a half-overwritten file. Worst case is a brief revert to version N followed by a re-pull. Corruption is structurally excluded.

N or N+1, never broken
03 / What syncs · what doesn't
IN SCOPE replicated by syncthing
  • projects/**/*.jsonl Conversation transcripts. Append-only — atomic appends sync cleanly without race.
  • memory/**/*.md Persistent memory files. Single-file atomic writes — exactly Syncthing's strength.
  • settings*.json User + local config. Small, infrequent edits. Conflict files are easy to resolve.
  • commands/ skills/ agents/ Custom workflow surface. Already in git — Syncthing keeps the working tree fresh between commits.
  • handoffs/ output-styles/ Session metadata. Small, append-style. Both machines benefit from current state.
EXCLUDED managed by another tool
  • *.db-wal · *.db-shm SQLite write-ahead-log + shared memory. Mutates continuously even when "idle." File-level sync produces torn pages.
  • beads/beads.db SQLite db itself. Already replicated by Dolt MVCC via bd dolt push/pull — that's its safe sync primitive.
  • .git/ Git's internal refs/index/packs are mid-mutation during commits. Replicated authoritatively via git push to GitHub.
  • logs/ High-frequency churn during active sessions. Saturates Syncthing's rescan budget — would never reach "converged."
  • cache/ · image-cache/ Derived data. Machine-specific paths. Regenerable on demand. Bandwidth waste with zero correctness benefit.
  • *.tmp · ~syncthing~tmp Sync staging files and editor scratch. Including them creates a sync-of-sync loop.
PRINCIPLE  ·  Don't double-manage state. Anything that has its own consistency protocol — SQLite WAL+checkpoint, Git refs+packs, build caches — gets excluded from file-level sync. The actively-synced surface is exclusively append-only logs and atomic single-file writes. That's the surface Syncthing handles flawlessly.

WHY IT WORKS  ·  git status doesn't ask "what changed?" — it computes sha1(working_tree_file) and compares to the index's last-committed blob hash. As long as both inputs are synchronized — Syncthing for the working tree, GitHub for the committed state — both machines independently arrive at the same diff. The two channels run in parallel; neither needs to know the other exists.

05 / The tunable knob
# ~/.config/syncthing/config.xml
fsWatcherEnabled = true
fsWatcherDelayS = 10

The default fsWatcherDelayS = 10 is what gives you the ten-second window. Drop it to 1 for lower latency, raise it to 60 to coalesce noisy projects. The trade is purely latency vs CPU/network churn.
